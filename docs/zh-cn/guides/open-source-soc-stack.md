# 构建开源SOC技术栈：Wazuh、TheHive、Cortex与MISP——拼装与集成的区别

有一套公认的免费开源SOC技术栈，多年来大致都是同样的四个名字：Wazuh负责检测，TheHive负责案件管理，Cortex负责可观测对象分析，MISP负责威胁情报。每个项目都确实擅长自己的工作，都久经实战检验，合在一起可以覆盖商业SOC套件所售卖的大部分能力。问题出在"合在一起"这个词上。工具本身很出色；它们之间的集成却是一个由你构建、然后由你长期维护的项目。

本指南将介绍每个组件的职责、把它们拼装起来的实际成本、当你为多个组织运营安全时需求会如何变化，以及SocTalk的定位——它位于这套技术栈*之上*，而不是取而代之。

## 经典的FOSS SOC技术栈

**[Wazuh](https://wazuh.com/)**是SIEM/XDR层：每个终端上部署一个代理，一个管理器对事件流应用检测规则，一个索引器（基于OpenSearch）存储并检索结果。它开箱即带文件完整性监控、漏洞检测、日志分析和一套庞大的默认规则集。告警在这里诞生。

**[TheHive](https://thehive-project.org/)**是案件管理层：一个安全事件响应平台，告警在这里转化为案件，案件承载任务和可观测对象，分析师团队在带有审计轨迹的环境中协作。如果说Wazuh是告警诞生的地方，TheHive就是调查生与死的所在。

**Cortex**是TheHive在可观测对象分析上的搭档。你把一个IP、哈希、域名或URL交给它，它的分析器插件会扇出到各类信誉和沙箱服务——VirusTotal、AbuseIPDB、Hybrid Analysis以及数十种其他服务——并带回一个裁决。它把"这是一个哈希"变成"这是全世界对这个哈希的认知"。

**[MISP](https://www.misp-project.org/)**是威胁情报平台：它跨情报源和共享社区聚合、关联并共享失陷指标。用MISP核查一个可观测对象，可以告诉你它是否属于某个已知的攻击活动或威胁行为体——这是其他三个工具本身都不具备的上下文。

四个工具，四项分工明确的职责，全部开源。纸面上看，这就是一个完整的SOC。

## 无人预算的集成税

这些工具每一个都能在一个下午装好。家庭实验室教程到此为止，而真正的工作才刚刚开始，因为它们没有一个能开箱即用地以生产级SOC所需的方式互相通信。

粘合层由你负责。Wazuh的告警不会自动变成TheHive的案件，你需要自己编写或采用一个转发器，并在两侧API变更时持续维护它。Cortex的分析器需要为每个提供商配置API密钥、处理速率限制，并决定哪种分析器处理哪种可观测对象类型。MISP需要配置情报源、调度同步任务，并在你敢于基于其自动化之前，先清理掉那些容易产生误报的指标。

然后是运维层面：四个产品意味着四套认证系统和API密钥轮换计划、四种可能在任意一次发布中破坏你粘合层的升级节奏、四套备份方案，而且——自从TheHive底层迁移到Cassandra/Elasticsearch之后——仅案件管理就需要一个不小的数据存储占用。再加上每一对组件之间的TLS、每个服务的监控，以及当Wazuh到TheHive的转发器悄无声息地停止转发时该由谁接收告警的问题。

这些都不是对工具本身的批评。这是组合独立项目的天性使然：集成层是第五个产品，只不过没有人为你发布它、编写文档或负责升级。

## 单一组织与MSSP：需求的分岔口

对单一组织而言，上述的税是付得起的。你只需构建一次技术栈，粘合层只服务一个租户，一名能干的工程师用兼职时间就能维持其健康运行。

对MSP或MSSP而言，需求会急剧分岔：

- **隔离没有商量余地。**客户A的告警、案件和指标必须对客户B可证明地不可见——这是合同要求，也常常是监管要求。共享的单租户工具让这变成一项逐工具的配置工作，并带来逐工具的故障模式。
- **每客户一套技术栈会让税成倍增长。**十个客户各用专属技术栈，意味着十套需要部署、升级和备份的Wazuh管理器与索引器——外加十份你的粘合层副本。
- **上线必须可重复。**第十一个客户应该是一条命令，而不是一周的wiki考古。手工搭建的技术栈会漂移；漂移会演变成事故。
- **单一视图。**覆盖二十个客户的分析师不可能在二十个仪表盘之间轮转。

这就是"FOSS SOC技术栈能用"与"FOSS SOC技术栈能作为一门生意运转"之间的鸿沟。

## SocTalk的定位：技术栈之上的控制平面，而非替代品

[SocTalk](https://github.com/soctalk/soctalk)并不替代这四个工具中的任何一个。它是一个Apache 2.0许可的多租户控制平面和AI分诊层，*围绕*这套技术栈构建，面向在自有Kubernetes上运行它的MSP和MSSP：

- **Wazuh是数据平面。**每个客户在隔离的命名空间中获得一套专属的Wazuh管理器和索引器，由控制平面负责置备——或者你也可以通过`provided`配置档接入已有的Wazuh。代理通过按主机名路由的入口注册，并使用租户级密钥。
- **AI分诊层位于Wazuh与你的分析师之间。**一个确定性的摄取漏斗在任何模型运行之前对告警进行去重、合并与关联；一个LangGraph代理式循环调查存活下来的告警；升级始终要经过人工审查关卡。详见[工作原理](/zh-cn/how-it-works)。
- **TheHive、Cortex和MISP是集成项**，在运行过程中被调用：Cortex提供可观测对象信誉，MISP提供威胁情报上下文，TheHive作为升级案件的导出目标。
- **多租户机制才是产品本身**：基于Cilium NetworkPolicy的命名空间隔离、作为数据兜底的Postgres行级安全（RLS）、租户生命周期状态机，以及按租户的LLM配置。

**请清楚了解V1的集成边界**，因为在这里诚实胜过营销：

- [TheHive导出](/zh-cn/integrate/thehive)是可选启用且**同步的**——工作进程在图节点执行时调用TheHive的API，创建案件和可观测对象。没有outbox，没有重试循环，也没有捆绑的TheHive子chart；如果TheHive不可达，失败会被记录，案件仅在SocTalk中继续推进。
- [Cortex](/zh-cn/integrate/cortex)在V1中**仅支持客户自管**——你自己运行Cortex，SocTalk调用它。没有捆绑的子chart；分析器选择使用一个硬编码映射，调用失败对本次运行是非致命的。
- **MISP**查询在流水线的`misp_worker`中针对你的MISP实例执行；捆绑的MISP子chart推迟到未来版本。
- **Slack**通知和双向审批代码存在于仓库中，但**尚未接入V1 chart运行时**——目前可用的人工在环（HIL）界面是仪表盘的审查队列。

换句话说：SocTalk打包了多租户Wazuh平面和分诊层，并*连接到*你自己运营的TheHive/Cortex/MISP实例。捆绑子chart的便利属于路线图，而非当前发布。

## 自建技术栈，还是部署SocTalk？

既然两条路都是开源的，这里给出诚实的判断标准：

**在以下情况自建四工具技术栈：**你是单一组织且有工程时间，你希望对每个组件拥有最大控制权，你的告警量对你的分析师人数而言尚可应付，且多租户与你无关。经典技术栈加上你自己的粘合层是一个经过验证的模式，而且每一根线都是你亲手焊的，你会理解其中的每一处细节。

**在以下情况考虑SocTalk：**你是MSP/MSSP，需要在一个控制平面之后可重复地为每个客户部署Wazuh技术栈、可证明的租户隔离，以及在分析师看到告警之前压缩告警量的AI分诊——而且你更愿意运维一个由Helm管理的平台，而不是N套手工搭建的技术栈。你仍然需要运行Kubernetes，并且在V1中，如果你需要TheHive、Cortex和MISP，仍然要自己运维它们。

最快的评估方式是[演示VM](/zh-cn/quickstart-vm)：一个镜像，一个浏览器向导，大约五分钟即可得到一个运行中的多租户安装，并完成一个演示租户的上线。在此基础上，[工作原理](/zh-cn/how-it-works)解释了流水线，而[TheHive](/zh-cn/integrate/thehive)和[Cortex](/zh-cn/integrate/cortex)页面准确记录了V1集成能做什么、不能做什么，方便你据此规划技术栈的其余部分。
